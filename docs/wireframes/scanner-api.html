Section 3: Scanner Submission API

The interface scanner partners use to submit scores. Does this payload structure and flow look right?

Auth: All scanner endpoints require Authorization: Bearer <api_key> header. Keys issued by CSA admin on scanner admission. Hashed with argon2 at rest.
Endpoints
POST /api/v1/scanner/scores Submit or update a score. Upserts on (service, scanner). Triggers aggregation recompute. GET /api/v1/scanner/scores List all scores submitted by this scanner. Supports pagination. GET /api/v1/scanner/scores/{score_id} Get a specific submission including match result and any CSA review flags. GET /api/v1/scanner/services List services this scanner has scored — useful for reconciliation and version tracking.

Response + Validation
201 Created / 200 Updated
{
  "status": "created" | "updated",
  "service_id": "uuid",
  "service_slug": "model/anthropic/claude/opus-4.6",
  "match_confidence": "high" | "medium" | "auto_created",
  "score_id": "uuid",
  "review_flagged": false,
  "published": true
}
Validation rules (422 on failure):
✓ All 6 pillar scores present, each in [0, 1000]
✓ composite_score within 5% of weighted pillar sum
✓ letter_grade matches composite_score band
✓ methodology_version is a known accepted value
✓ service.type matches scanner's admitted service_types[]
✓ hosting_type = "hosted" enforced for mcp_server in v1
1.Check service_identifiers for any submitted external ID → high confidence, auto-link 2.Else: match on identity tuple → medium confidence, auto-link + flag for CSA review 3.Else: auto-create new service listing → auto_created, flag for CSA review 4.Score always published immediately. CSA review never blocks publication. 5.Recompute aggregated_scores for the affected service.
